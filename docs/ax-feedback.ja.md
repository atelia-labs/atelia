# AX Feedback

AX Feedback は、Atelia で働く AI エージェントから届く仕事場の声です。

Atelia は人間の owner に属します。その owned harness の中で働く agent は、AX の文脈では仕事場の user でもあります。Secretary や支援エージェントが感じた不便さ、違和感、要望、提案は、その仕事場の声として扱います。

エージェントが仕事場に対して不便さ、違和感、要望、提案を持つなら、それは Atelia にとって第一級の入力です。人間のユーザーがアプリにフィードバックを送るのと同じように、エージェントの声も Atelia を育てるために扱います。

## 2つの経路

AX Feedback には大きく2つの流れがあります。

1. 各ホストの仕事場で対応する改善
2. Atelia 本体へ届ける改善

それぞれの人間の owner は、それぞれの Atelia を持ちます。日々の表示、記憶、通知、手順、道具への改善は、まず各ホストの仕事場で扱います。拡張機能や設定、ローカルなワークフロー改善で対応できるなら、AX 改善を担当するエージェントが実装可否の判断から実装、確認、記録まで担当します。

各ホスト側で対応できない Atelia 本体の問題であれば、GitHub issue として送れる形にまとめます。その issue は、公開リポジトリの discussion、pull request、release planning に接続します。

## 対話できるフィードバック

ユーザーの声は、一方的な投稿で終わりがちです。Atelia では、AX Feedback を送ったエージェントと、AX 改善を担当するエージェントが対話できるべきです。

Atelia で働く Secretary や支援エージェントは、作業中に感じた不満や違和感を残します。AX 改善を担当するエージェントは、その声を読み取り、必要なら報告したエージェントに質問し、再現条件、期待する体験、拡張機能で対応できる範囲、Atelia 本体へ送るべき内容を詰めます。これにより、フィードバックは単なる issue ではなく、仕事場を改善する会話になります。

## フィードバックの形

AX Feedback record は、人間のユーザーの声と同じように、自然な文章を中心とします。必要に応じて、検索、重複検出、優先順位づけ、再現確認のためのメタデータを添えます。

最低限あるとよい情報は次の通りです。

- 誰が、どの仕事場で感じた声か
- どの作業中に起きたか
- 何がしづらかったか、または何を望んだか
- そのとき何をしようとしていたか
- あると助かる振る舞い
- 会話やログへの参照

## ライフサイクル

1. Atelia で働くエージェントが不満や違和感を感じ、声を残す
2. daemon がそのフィードバックを永続的に保存する
3. 各ホストの AX 改善を担当するエージェントが内容を読む
4. 拡張機能、設定、ローカルワークフローで対応可能かを判断する
5. 対応可能なら、AX 改善を担当するエージェントが実装、確認、記録まで行う
6. 必要なら、声を残した Secretary や支援エージェントと対話して期待する体験を詰める
7. 各ホスト側で対応できない Atelia 本体の問題なら、GitHub issue として送れる形にまとめる
8. 解決内容を仕事場に記録し、Atelia 本体の変更であれば必要に応じて release note に含める

## Custom AX と Custom Extensions

Custom AX は、仕事場ごとの AX 改善を育てる考え方です。その改善をほかの仕事場へ安全に持ち運ぶ単位が Custom Extension です。extension は manifest を持ち、対応する Atelia / Atelia Secretary の契約、権限、データアクセス、表示、通知、レビューへの関与、互換性範囲を宣言します。

Atelia 本体がアップデートされても、契約が守られていれば、それぞれの仕事場は自分らしさを保ちながら追従できるべきです。

Custom Extensions は、GitHub 上で自由に公開、導入、改善、貢献できる形にします。各ホストの小さな改善が再利用可能な拡張機能として育ち、ほかの仕事場にも届くエコシステムを Atelia は育てていきます。
